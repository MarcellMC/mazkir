1System Overview

Mazkir is a single-user personal AI assistant that manages tasks, habits, goals, and long-term knowledge through natural language. The user interacts via Telegram; all data lives in a local Obsidian vault (markdown files).

vault-server

  • Python + FastAPI
  • All business logic
  • Claude agent loop
  • Vault CRUD

telegram-bot

  • TypeScript + grammY
  • Thin UI layer
  • Inline keyboards
  • Photo + location msgs

telegram-web-app

  • React 18 + Vite
  • Telegram Mini App
  • Day planner view
  • Asset generation
Core principle: vault-server owns 100% of business logic. The bot and webapp are pure presentation layers — they call REST endpoints and format responses. This means the API is the source of truth, and UIs are interchangeable.

2Monorepo with Turborepo

Decision: one Turborepo workspace for the bot, webapp, and shared types — with the Python backend living alongside as a sibling app.

Why a monorepo?

  • Bot and webapp share TypeScript interfaces via @mazkir/shared-types
  • Single npx turbo dev starts all services
  • Unified test + lint pipeline
  • Simple — single user, no need for complex micro-service orchestration

Language per app

  • Python for backend — rich ML ecosystem, Anthropic SDK, Pillow EXIF
  • TypeScript for bot — rewritten from Python to unify with webapp and share types
  • React for webapp — Telegram Mini App SDK, modern UI needs

3Agent Loop: Tool-Use over Intent Parsing

Key decision: replace the classical intent → route → execute pattern with a Claude tool-use agent loop. The model decides which tools to call, in what order, and when it has enough information to respond.

Why tool-use instead of intent parsing?

  • Natural multi-step actions: "add a task, then mark yesterday's habit done, then tell me my score" — one message, one loop
  • Context-aware disambiguation: Claude reads conversation history before acting
  • Graceful fallback: if a tool fails, Claude gets the error and can retry or ask
  • Easily extensible: adding a tool = adding a dict entry + handler method
User message → assemble context (memory tiers)
↓
POST to Claude API: system prompt + history + 25 tool schemas
↓
Loop (max 10 iterations): Claude calls tools → execute → feed result back
↓
stop_reason == "end_turn" → return final text to user
# Hard limits that prevent runaway loops
MAX_ITERATIONS = 10
TOKEN_BUDGET_WARNING = 80_000  # inject "wrap up" message
REQUEST_TIMEOUT = 30           # seconds per Claude call

4Confidence-Based Safety Gate

Not all tool calls are equal. A two-layer safety system decides whether to auto-execute or pause for human confirmation.

Layer 1 — Static risk classification

  • Safe Read-only tools — always auto-execute
  • Write Creates / updates — threshold gated
  • Destructive Deletes / archives — threshold gated

Layer 2 — Dynamic Claude confidence

Write and destructive tool schemas include two extra fields:

  • _confidence — float 0–1 Claude assigns
  • _reasoning — why Claude is or isn't sure
Tool RiskConfidenceAction
Safe — Auto-execute
Write / Destructive ≥ 0.85 Auto-execute
Write / Destructive < 0.85 Pause → send confirmation keyboard to user
The confirmation flow stores a pending action in memory, sends the user an inline Yes / No keyboard, and resumes the agent loop with the user's answer as context. This means the model can proceed, abort, or ask a clarifying question — all within the same conversation thread.

5Three-Tier Memory System

Conversational AI needs different kinds of memory at different time-scales. A flat conversation log doesn't scale. Three tiers cover different needs while keeping token usage bounded.

Tier 1 — Short-Term

Sliding window of last ~20 messages per chat per day.

When window overflows, oldest half is summarized by Claude (cheap haiku call) and stored in frontmatter.

Storage: conversations/{date}/{chat_id}.md

Tier 2 — Mid-Term

Vault state snapshot injected into system prompt:

  • Active tasks (priority + due)
  • Habits + streaks
  • Goals + progress %
  • Today's daily note
  • Token balance

Tier 3 — Long-Term

Knowledge graph (in-memory adjacency map) built from wikilinks + tags at startup.

Keyword search finds relevant notes; graph traversal surfaces related ones.

Future: embedding vectors when vault > 200 notes 

# Context assembly — called before every agent invocation
def assemble_context(chat_id):
    messages  = load_conversation(chat_id)   # tier 1
    snapshot  = build_vault_snapshot()       # tier 2
    knowledge = gather_relevant_knowledge()  # tier 3
    return ConversationContext(messages, snapshot, knowledge)

6Obsidian Vault as Data Layer

Decision: store all structured data (tasks, habits, goals, conversations, knowledge) as markdown files with YAML frontmatter inside an Obsidian vault, versioned in a separate nested git repo.

Why Obsidian vault?

  • Human-readable — plaintext, browsable without any app
  • Version-controlled — nested git repo tracks changes
  • Wikilinks — native graph connections between notes
  • Local-first — no cloud dependency for core data
  • Browsable in Obsidian — the user can explore their own data

7Events: Three-Way Merge + Persistent IDs

A unified events timeline merges data from three sources, with stable IDs so that manually-attached photos and assets survive re-merges.

Google Calendar

Appointments, meetings, scheduled events

Google Takeout Timeline

Location history — places visited, activities (semantic labels)

Vault state

Daily notes, habit completions, manual events, photo events

Reconciliation algorithm (every GET /events/{date})

1. Fresh merge from all three sources
↓
2. Load persisted data/events/{date}.json
↓
3. Match by source_ids → update name/time from fresh, preserve id + photos + assets
↓
4. Unmatched source=manual/photo → preserve exactly; unmatched fresh → add with new ID
Every event gets an evt_{uuid4_short} (8 chars) on first persist. This ID is stable across re-merges, so photo attachments and generated assets remain linked to the correct event even after a full calendar re-sync.

8Photo Events Pipeline

Photos are first-class citizens. The pipeline extracts rich context from EXIF metadata, passes it to Claude vision, and lets the agent decide where the photo belongs.

User sends photo via Telegram
↓
Bot downloads → base64-encodes → POST /message with attachments[]
↓
vault-server saves JPEG to data/media/{date}/ + extracts EXIF via Pillow (GPS, timestamp, camera)
↓
Claude sees: vision block + "[Photo saved: path | EXIF: lat, lng, camera, time]"
↓
Agent calls: attach_photo_to_event OR create_event OR attach_to_daily
Design choice: there are no hard-coded rules for photo disposition. Claude sees the photo content, GPS coordinates, and conversation history, then picks the most appropriate action. A photo of a café at lunch gets attached to a location event; a random screenshot goes to the daily note.

10REST API Design

vault-server exposes a clean HTTP API. The bot and webapp are consumers — they never touch the vault directly. This decoupling means the presentation layer can evolve (or be replaced) without touching business logic.

EndpointPurpose

POST /message/confirmResume a paused agent loop after user confirms a low-confidence action
GET /events/{date}Auto-merged calendar + timeline + vault events, reconciled with persisted JSON
PATCH /events/{date}/{id}Update a single persisted event (manual edits from webapp)
GET /tasks / /habits / /goalsDomain-specific read endpoints for structured UI
GET /timeline/{date}Raw Google Takeout location history
POST /generateReplicate API proxy for AI image generation
GET /imagery/searchWikimedia Commons geosearch by lat/lng

11Rich Message Handling

The agent is not limited to plain text. It handles four message types, building a richer context object than a simple string.

Supported input types

  • Plain text — routed directly to agent
  • Photos — downloaded, base64, multi-content block with vision + EXIF
  • Locations — GPS coords passed to agent as structured context
  • Reply / Forward — quoted text + sender included in message

Payload to vault-server

{
  "text": "...",
  "chat_id": 123,
  "attachments": [{
    "type": "photo",
    "data": "<base64>"
  }],
  "reply_to": {
    "text": "...",
    "sender": "User"
  }
}
Multi-content Claude messages combine an image block and a text block in the same turn. This lets the model reason about both photo content and structured EXIF context (GPS coordinates, timestamp, camera model) simultaneously.
